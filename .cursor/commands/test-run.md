# テスト・ビルド一括実行

## 概要

アプリの品質とデプロイ可否を確認するため、型チェック・リント・テスト・ビルドを順に実行する。
**テスト実行とレポート作成に徹し、コード修正の提案はレポートに記載して報告する。**
以下は TypeScript プロジェクトを想定したコマンド例である。
プロジェクトの `package.json` に合わせて読み替えること。

## 手順

### 1. 開発サーバー起動

```sh
bun run dev
```

満たすべき基準

- サーバーが起動し、指定ポートで待ち受けていること

### 2. コード品質チェック

型チェック・リント・フォーマットチェックを実行する。フォーマットの自動適用は、ユーザーから明示的に依頼された場合のみ行う。

```sh
# 型チェック（コンパイルは行わない）
bunx tsc --noEmit

# リント
bun run lint

# フォーマットチェック（自動適用は依頼時のみ bun run format）
bun run format:check
```

満たすべき基準

- `tsc --noEmit` で型エラーが 0 件であること
- `lint` が警告・エラーなしで完了すること
- `format:check` でフォーマット差分がないこと（または依頼に応じて `format` 適用後に再チェックが通ること）

### 3. テスト実行

ユニット・統合・E2E の順で実行する。
プロジェクトに `test:all` がある場合は一括実行も可。

```sh
# ユニットテスト
bun run test:unit

# 統合テスト
bun run test:integration

# E2E テスト（開発サーバー起動済みであること）
bun run test:e2e

# 一括実行（定義されている場合）
bun run test:all
```

単体ファイル・ディレクトリ指定で実行する例:

```sh
bun test src/tests/unit/AuthButton.test.tsx
bun test --verbose src/tests/
```

満たすべき基準

- ユニット・統合・E2E のいずれも、実行した分について全件成功（exit code 0）であること
- スキップ・pending はレポートに明記すること
- 失敗がある場合は、失敗ケース名・エラー内容をレポートに含めること

### 4. カバレッジ

カバレッジを取得し、閾値チェックやレポート確認を行う。
目標割合はプロジェクトごとに異なる。

```sh
# カバレッジレポート生成
bun run test:coverage

# 閾値チェック（CI などで利用）
bun run test:coverage:check

# ブラウザでレポートを開く（任意）
bun run test:coverage:open

# バッジ生成（任意）
bun run test:coverage:badge
```

満たすべき基準

- プロジェクトで定義されたカバレッジ閾値（行・分支など）を満たしていること
- 未達の場合は、どのファイル・どの範囲が未カバーかをレポートに記載すること

### 5. ビルド検証

本番ビルドが通ること、および起動確認を行う。

```sh
# 本番ビルド
bun run build

# 本番サーバーでローカル起動（任意）
bun run start
```

満たすべき基準

- `build` が警告・エラーなしで完了すること
- ビルド成果物（例: `.next`, `dist`）が期待どおり出力されていること
- `start` まで行う場合は、起動後すぐに異常終了しないこと

### 6. テストレポートの作成と報告

実行結果をまとめ、報告する。

- 実施した手順（2〜5 のうち実行したもの）と、各ステップの成否
- 失敗・スキップしたテストの名前と理由・エラー内容
- カバレッジ未達がある場合は対象ファイルと範囲
- コード修正の提案がある場合は、ここに箇条書きで記載する

満たすべき基準

- 実行したコマンドと結果が対応して分かること
- 失敗や未達がある場合、再現・修正に必要な情報が含まれていること
- 修正提案は具体的（ファイル名・関数名・方針）であること
