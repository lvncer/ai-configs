# 大規模 PR レビュー（Commit駆動）

## 概要

変更行数が多い PR（目安: 1000行以上、または10commit以上）を
**commit単位で分解し、動作変更リスクを中心にレビューする。**

通常の一括diffレビューは行わず、
**履歴ベース（縦方向）でレビューする。**

実行時は **GitKraken MCP を最優先** で使用し、未対応操作のみ `gh` をフォールバックする。

## 最重要ルール

- AIはレビュー結果を **先に人間へ報告** する
- 人間の明示指示があるまで、PRへ書き込まない
- 「修正案の提示」と「PR投稿」は分離する
- まずcommit構造を把握し、危険commitを特定する

## 手順

### 1. PR情報とcommit一覧を取得

#### MCP優先

- `pull_request_get_detail`
- commit一覧取得
- 必要に応じて個別commit差分取得

#### フォールバック

```bash
gh pr view <PR_NUMBER> --json title,body,commits
gh pr diff <PR_NUMBER>
```

### 2. Commit構造の分析（必須）

各commitについて以下を整理する：

- 変更目的（推測）
- 主な変更カテゴリ
- 動作変更リスク（High / Medium / Low）

#### 変更カテゴリ分類

- 構造変更（ファイル分割・移動）
- 型強化（型ガード追加・型定義変更）
- 純粋リネーム
- ロジック変更
- テスト追加 / 更新
- 依存関係変更

### 3. Commitストーリー要約（最初に出力）

例：

```md
このPRは以下の流れで構成されている：

1. 型定義整理
2. サービス層の分割
3. validationロジック修正
4. テスト追加
```

### 4. 危険commitの特定

以下を含むcommitは精査対象：

- 条件分岐の変更
- return値の変更
- async化
- 例外処理変更
- 型定義の破壊的変更
- public API変更

それ以外（純粋リネーム・機械的整形など）は軽量確認。

### 5. Commit単位レビュー

各危険commitについて確認：

#### ロジック非変更保証（リファクタ前提）

- 条件式の意味が変わっていないか
- 副作用の順序が変わっていないか
- 例外条件が増減していないか
- silent failureが発生していないか

#### 型ガード追加時

- narrowingが過剰になっていないか
- any化していないか
- ガード漏れがないか

#### ファイル分割時

- 循環依存がないか
- 公開APIが変わっていないか
- レイヤ境界が明確か

#### テスト追加時

- 仕様固定として十分か
- 失敗時に原因追跡可能か
- ロジック変更commitと整合しているか

## レビュー結果の報告フォーマット

```md
## レビュー結果（人間向け）

### PR構造サマリ

| #   | commit概要 | 種類 | 動作変更リスク |
| --- | ---------- | ---- | -------------- |
| 1   |            |      |                |

### 総評

- このPRの設計的評価

### 必須修正（Critical/High）

1. [severity] 問題の要約
   - 対象commit:
   - 根拠:
   - 影響:
   - 修正案:

### 改善提案（Medium/Low）

1. 内容
   - 理由:
   - 提案:

### PR への投稿候補文

- review body:

  <そのまま投稿できる文章>

- inline comment 候補:
  - <file/path>:<line付近> コメント案
```

## 運用ポリシー（大規模PR特化）

- 差分全体を均一密度でレビューしない
- 危険commitにレビュー密度を集中する
- 機械的変更には時間を使わない
- 「意味が変わっていないか」を最優先で確認する
