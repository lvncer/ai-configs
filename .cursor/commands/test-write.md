# テスト作成

## 概要

新規・既存機能に対するテストを、プロジェクトの規約に沿って追加する。
ユニット・統合・E2E のどれを書くかは対象に応じて選び、配置・命名・中身の基準を満たすこと。

## 手順

### 1. テスト種別の選定

対象に応じてユニット・統合・E2E のいずれか（または組み合わせ）を選ぶ。

| 種別         | 向いている対象                                     | 特徴                                     |
| ------------ | -------------------------------------------------- | ---------------------------------------- |
| **ユニット** | 単一関数・コンポーネント・純粋ロジック             | 依存をモックし、高速・隔離して検証       |
| **統合**     | API・DB・外部サービス連携                          | 実モジュールを組み合わせた振る舞いを検証 |
| **E2E**      | 画面遷移・フォーム送信・クリティカルなユーザー操作 | ブラウザから実際の UI を操作して検証     |

満たすべき基準

- 1 テストファイルが「1 つの対象（1 ファイル or 1 機能）」に紐づいていること
- 重複する検証は避け、ユニットで十分なものは E2E に寄せすぎないこと

### 2. 配置と命名

プロジェクトの既存テスト構成に合わせて配置する。
例は TypeScript / 一般的な構成。

- **配置例**
  - ユニット: `src/tests/unit/` または `src/**/__tests__/` / `*.test.ts(x)` の併存
  - 統合: `src/tests/integration/` または `tests/integration/`
  - E2E: `e2e/` または `tests/e2e/`

- **命名**
  - 対象ファイルが `AuthButton.tsx` なら、`AuthButton.test.tsx` または `AuthButton.spec.tsx`
  - 機能単位なら `auth-login.test.ts` のようにケバブケースも可
  - 既存の `*.test.*` / `*.spec.*` のどちらかに揃える

満たすべき基準

- 既存のテストディレクトリ・命名規則と一致していること
- ファイル名から「何のテストか」が分かること

### 3. テストファイルの構成

describe / it（または test）でスコープを分け、1 ケース 1 アサーションに近づける。

```ts
// ユニット例（Bun / Vitest 風）
import { describe, it, expect, beforeEach } from "bun:test";
import { render, screen } from "@testing-library/react";
import { AuthButton } from "@/components/AuthButton";

describe("AuthButton", () => {
  beforeEach(() => {
    // 共通セットアップ
  });

  it("未ログイン時に「ログイン」を表示する", () => {
    render(<AuthButton user={null} />);
    expect(screen.getByRole("button", { name: /ログイン/ })).toBeInTheDocument();
  });

  it("ログイン時に「ログアウト」を表示する", () => {
    render(<AuthButton user={{ name: "test" }} />);
    expect(screen.getByRole("button", { name: /ログアウト/ })).toBeInTheDocument();
  });
});
```

満たすべき基準

- 最上位の `describe` が「対象モジュール or 機能」を表していること
- 各 `it` の説明文が「条件 → 期待する結果」を日本語で簡潔に表していること
- セットアップは `beforeEach` / `beforeAll` にまとめ、テスト本体が読みやすいこと

### 4. モックとフィクスチャ

外部依存はモックし、再現性のあるデータはフィクスチャとして用意する。

- **モック**: API クライアント・ルーター・グローバル関数などは、テスト内で差し替え可能にし、期待する呼び出しを検証する
- **フィクスチャ**: ユーザーオブジェクト・API レスポンスなどは `fixtures/` やテストファイル直下の定数で共有し、変更時は一箇所だけ直せばよいようにする

満たすべき基準

- テストがネットワーク・DB・時刻に依存せず、どこでも同じ結果になること
- モックの期待値（引数・呼び出し回数）がテストの意図と一致していること

### 5. アサーションとエッジケース

期待値は `expect` で明示し、正常系に加えて境界・異常系も 1 つ以上入れる。

- 正常系: 典型的な入力で期待する出力・副作用
- 境界: 空文字・0 件・最大長・undefined など
- 異常系: エラー時・不正入力時のメッセージやステータス

満たすべき基準

- 各 `it` で「何を検証しているか」がアサーションから分かること
- 対象が分岐やエラー処理を含む場合、その経路を少なくとも 1 ケースでカバーしていること

### 6. 作成後の確認

作成・修正したテストだけを実行し、すべて成功することを確認する。その後、必要に応じて「テスト・ビルド一括実行」で全体を流す。

```sh
# 追加したファイルだけ実行
bun test src/tests/unit/AuthButton.test.tsx

# 対象ディレクトリを指定
bun test src/tests/unit/

# 一括実行は run-all-test を参照
bun run test:unit
```

満たすべき基準

- 追加・変更したテストがすべてグリーンであること
- 既存の他テストを壊していないこと（必要なら `test:unit` / `test:all` で確認）
- プロジェクトにカバレッジ閾値がある場合は、新規コードが閾値を下げないこと
